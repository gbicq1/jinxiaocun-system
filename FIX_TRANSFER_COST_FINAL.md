# 调拨单成本价获取逻辑最终修正

## 问题描述

调拨单获取的成本价与成本结算明细显示的最后一笔业务的库存结余成本价不一致。

### 问题表现

- **调拨单获取的成本价**：¥31.45（错误）
- **成本结算明细最后一笔业务的成本价**：¥25.50（正确）
- **加权平均单价**：¥22.15（不是调拨单应该使用的价格）

### 成本结算明细截图分析

```
期初：8 个，成本 ¥11.00/个，金额 ¥88
出库：3 个，成本 ¥11.00/个
入库：5 个，成本 ¥40.00/个，金额 ¥200  ← 最后一笔入库业务
出库：3 个，成本 ¥11.00/个

库存结余变化：
1. 期初：8 个，¥11.00/个，¥88
2. 出库 3 个后：5 个，¥11.00/个，¥55
3. 入库 5 个后：10 个，¥25.50/个，¥255  ← 最后一笔业务后的成本价
4. 出库 3 个后：7 个，¥25.50/个，¥178.50

加权平均单价计算：
(期初¥88 + 入库¥200) / (8 + 5) = ¥288 / 13 = ¥22.15
```

**关键发现**：
- 加权平均单价 ¥22.15 是整个月的平均成本
- 最后一笔业务后的成本价 ¥25.50 是调拨单应该使用的价格

## 解决方案

### 核心逻辑

**动态计算调拨日期所在期间的最后一笔业务的库存结余成本价**。

计算步骤：
1. 获取当月所有库存单据（采购入库、销售出库）
2. 按日期顺序排序
3. 逐笔计算每笔业务后的库存结余数量和金额
4. 返回最后一笔业务后的成本价（库存金额 / 库存数量）

### 实现代码

```typescript
export const getCostFromSettlement = (productId, warehouseId, transferDate) => {
  // 1. 获取当月所有库存单据
  const allRecords = getAllInventoryRecordsForMonth(year, month, productId, warehouseId)
  
  // 2. 按日期排序
  allRecords.sort((a, b) => new Date(a.date) - new Date(b.date))
  
  // 3. 逐笔计算库存结余
  let runningQty = 0
  let runningAmount = 0
  
  for (const rec of allRecords) {
    if (rec.type === 'inbound') {
      // 入库：增加数量和金额
      runningQty += rec.qty
      runningAmount += rec.amount
    } else if (rec.type === 'outbound') {
      // 出库：减少数量，按当前成本价减少金额
      const currentCostPrice = runningQty > 0 ? runningAmount / runningQty : 0
      runningQty -= rec.qty
      runningAmount -= rec.qty * currentCostPrice
    }
  }
  
  // 4. 返回最后一笔业务后的成本价
  const finalCostPrice = runningQty > 0 ? runningAmount / runningQty : 0
  return Number(finalCostPrice.toFixed(2))
}
```

### 计算示例

```
业务记录（按日期排序）：
1. 期初：8 个，¥11.00/个，¥88
2. 销售出库：3 个
3. 采购入库：5 个，¥40.00/个，¥200
4. 销售出库：3 个

计算过程：
1. 期初后：qty=8, amount=¥88, cost=¥11.00
2. 出库 3 个后：
   - 当前成本价：¥88/8 = ¥11.00
   - qty=8-3=5, amount=¥88-3×¥11=¥55, cost=¥11.00
3. 入库 5 个后：
   - qty=5+5=10, amount=¥55+¥200=¥255
   - cost=¥255/10=¥25.50  ← 最后一笔业务后的成本价
4. 出库 3 个后：
   - 当前成本价：¥25.50
   - qty=10-3=7, amount=¥255-3×¥25.50=¥178.50, cost=¥25.50

返回结果：¥25.50 ✅
```

## 性能考虑

### 优化措施

1. **只获取当月数据**：不遍历全年数据
2. **只获取匹配的产品和仓库**：减少数据处理量
3. **单次遍历计算**：O(n) 时间复杂度
4. **缓存机制**：可以添加缓存避免重复计算

### 性能对比

| 指标 | 复杂实现（第一次） | 简单实现（第二次） | 动态计算（本次） |
|------|------------------|------------------|----------------|
| 时间复杂度 | O(n²) | O(m) | O(k) |
| 数据范围 | 所有单据 | 结算记录 | 当月单据 |
| 操作次数 | 250,000 | 15 | ~10 |
| 响应时间 | 卡顿 | <1ms | <1ms |
| 准确性 | ✅ | ❌ | ✅ |

说明：
- n = 所有单据数量（~500）
- m = 结算记录数量（~15）
- k = 当月单据数量（~10）
