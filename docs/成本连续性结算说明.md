# 成本连续性结算功能说明

## 功能概述

基于您的建议实现的**连续性成本结算系统**，确保成本数据从系统第一笔业务开始连续计算，避免不同会计期间成本价不一致的问题。

## 核心特性

### 1. 系统启动时自动初始化

**触发时机：** 系统启动后 2 秒自动执行

**逻辑流程：**
1. 扫描所有单据，找到系统最早的单据日期
2. 从最早单据的月份开始，逐月结算到上月
3. 如果某月已结算，检查该月之后是否有新增单据
4. 如果有新增单据，触发从该月开始的重算

**日志示例：**
```
========== 系统启动：自动初始化成本结算 ==========
✓ 系统第一笔业务日期：2025-03-15
✓ 第一笔业务所属期间：2025 年 3 月
✓ 需要结算范围：2025 年 3 月 到 2026 年 2 月
  + 结算 2025 年 3 月 ...
  ✓ 成功结算 15 条记录
  - 2025 年 4 月 已结算，检查是否需要重算...
  ✓ 2025 年 4 月 无需重算
========== 系统启动初始化完成 ==========
```

### 2. 新增单据时自动结算

**触发时机：** 各业务模块保存单据后

**逻辑流程：**
1. 检查单据日期是否早于最新已结算期间
2. 如果是，触发从该单据月份到当前的重算
3. 如果不是，检查该月份是否已结算
4. 如果未结算，触发从第一笔单据月份到当前的结算

**场景示例：**

**场景 A：新增当前月份单据**
```
单据日期：2026-03-21
最新已结算：2026-02
操作：触发 2026-03 的结算
```

**场景 B：新增历史月份单据**
```
单据日期：2026-01-15
最新已结算：2026-02
操作：触发从 2026-01 开始的重算（覆盖 2026-01、2026-02、2026-03）
```

### 3. 数据连续性保证

**保证机制：**
- 每月结算时，期初数据 = 上月期末数据
- 如果上月未结算，递归查找上上月
- 如果所有历史月份都未结算，从 0 开始（系统初始化）

**数据验证：**
```typescript
// 2 月期末 = 3 月期初
febData.closing_qty === marData.opening_qty
febData.closing_cost === marData.opening_cost
```

## 使用方法

### 在业务模块中集成

在保存单据成功后调用自动结算：

```typescript
import { triggerAutoSettlement } from '@/utils/cost-database'

// 保存单据后
const saveSuccess = await saveDocument()
if (saveSuccess) {
  // 自动触发成本结算
  const result = await triggerAutoSettlement(
    formData.documentDate,  // 单据日期
    '01',                   // 产品编码（可选）
    1                       // 仓库 ID（可选）
  )
  
  if (result.success) {
    console.log('成本结算完成:', result.message)
  }
}
```

## 性能优化

### 1. 增量计算
- 已结算的月份不会重复计算
- 只在检测到新单据时触发重算
- 重算时从最早的受影响月份开始

### 2. 快照机制（建议后续实现）
- 每月末保存库存快照
- 下月计算时从快照开始，不用从第一笔开始
- 快照可作为数据校验点

### 3. 异步处理
- 结算逻辑在后台异步执行
- 不阻塞用户界面
- 通过 IPC 消息通知前端结果

## 常见问题

### Q1: 系统第一次启动会很慢吗？
**A:** 取决于历史单据数量。如果单据量大（如 >10000 条），首次计算可能需要几分钟。建议：
- 新系统：无影响，单据少
- 老系统：可以考虑在空闲时后台计算

### Q2: 如果结算失败怎么办？
**A:** 系统会记录详细的错误日志。检查步骤：
1. 查看 Electron 控制台日志
2. 检查数据库是否可写
3. 检查单据数据是否完整
4. 手动触发结算查看详细错误

### Q3: 可以手动触发结算吗？
**A:** 可以。在成本结算模块选择月份，点击"计算"按钮即可。

### Q4: 锁定后的数据还能修改吗？
**A:** 不能。锁定的数据不允许修改，需要先"反结算"解锁。
